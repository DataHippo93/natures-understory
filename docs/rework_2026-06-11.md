# Understory Rework — 2026-06-11

## What changed and why

### Root causes found
1. **Silent fake data.** `lib/data.ts` fell back to *synthetic demo data* on any
   error. Since the Clover warehouse tables were dropped (2026-05-01), the
   Shifts DOW breakdown and parts of reports were quietly showing made-up
   numbers. This is the "Homebase and Clover sales data is not right."
2. **Dead Clover sync.** `/api/cron/daily-sync` ran every morning against
   dropped tables and failed every day since May 1.
3. **Reports queried dropped tables** (`sales_line_items` etc.).
4. **`compute-features` cron** also read dropped tables, and its old REST pull
   was silently capped at 1000 rows.
5. **Personal credentials committed** in `e2e/auth.test.ts`,
   `scripts/create-user.mjs`, and quoted in an audit doc.

### Fixes
- Sales reporting now reads the **Thrive warehouse** (`thrive_sales_history` +
  `thrive_product_catalog`) via the new `lib/thrive.ts`. Reports show
  department/item revenue, units, **margin**, and a "data through" badge.
- Labor ratio: daily sales denominators from Thrive; labor from Homebase live.
- Shifts: hourly analysis stays on Clover's live Payments API (only hourly
  source), aggregation logic unchanged.
- **Demo data only with `DEMO_MODE=true`.** Errors now surface via the new
  `app/error.tsx` boundary ("Data Unavailable") instead of fake numbers.
- Dead Clover sync routes stubbed to **410 Gone** (delete directories when git
  is available): `app/api/cron/daily-sync`, `app/api/sync/{categories,items,sales}`,
  `app/api/admin/enrich-categories`. Components `sync-button.tsx` and
  `sync-panel.tsx` are now unreferenced — delete too. Also delete
  `app/api/debug/category-test` (hits Clover with hardcoded April dates).
- `compute-features` ported to Thrive with SQL-side aggregation.
- Credentials scrubbed from current files (env-var required now). **Rotate the
  exposed password** — it remains in git history.

### New: Cooler Monitor (`/coolers`)
- `cooler_config` + `cooler_readings` tables (migration applied to prod +
  saved as `supabase/migrations/007_cooler_monitoring.sql`).
- `/api/cron/pull-coolers` polls Home Assistant `/api/states` every 5 min
  (vercel.json cron added), auto-discovers cooler/freezer temperature sensors,
  default ranges 32–41°F (coolers) / -20–5°F (freezers), editable per-sensor
  in `cooler_config`.
- Dashboard flags **alert** when continuously out of range ≥30 min, plus
  warning/stale/no-data states, 24h sparklines, and an alert strip on the
  home dashboard.
- Status logic is pure + unit-tested (`__tests__/coolers.test.ts`).
